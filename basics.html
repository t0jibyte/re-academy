Transistors, Logic Gates & Binary

Every single operation your CPU performs comes down to transistors — tiny electronic switches that are either ON (1) or OFF (0). A modern CPU has billions of them. This two-state nature is why all computer data is binary: 1s and 0s.

Transistors are grouped into logic gates that implement boolean logic. There are three fundamental ones — AND, OR, NOT — and every other circuit can be built from them. XOR is especially important: it's used everywhere from checksums to encryption.

From gates come adders, multiplexers, flip-flops (1-bit memory cells), registers, and full CPUs. The entire stack — from your C code down to hardware — is just logic gates.

Number Systems — Decimal, Binary, Hexadecimal

You need three number systems. Decimal (base-10) is what humans use every day. Binary (base-2) is how the CPU actually stores and manipulates values. Hexadecimal (base-16) is what you use when reading memory — one hex digit = exactly 4 bits, so one byte = two hex digits. Every memory address, opcode, and flag value you see is in hex.

Bits, Bytes, and Data Sizes

All data in memory is a sequence of bytes. The CPU can operate on 8, 16, 32, or 64 bits in a single instruction. Knowing the exact size of every type is critical — a size confusion is often the root cause of security bugs.

Two's Complement — How Negative Numbers Work

Every CPU uses two's complement to represent signed integers. The most-significant bit is the sign bit (0=positive, 1=negative). To get the negative of a number: flip all bits, then add 1. The beauty of two's complement is that addition and subtraction work identically for signed and unsigned numbers — the CPU doesn't need separate hardware.

two's complement — 8-bit examples// How -5 is stored:
Step 1: start with +5  →  0000 0101
Step 2: flip all bits  →  1111 1010
Step 3: add 1          →  1111 1011  = -5

// Verify: 5 + (-5) should equal 0
  0000 0101   (+5)
+ 1111 1011   (-5)
-----------
  0000 0000   (= 0, carry discarded) ✓

// Key values in 8-bit signed:
 0x7F = 0111 1111 =  127  (max positive)
 0x80 = 1000 0000 = -128  (most negative — cannot negate!)
 0xFF = 1111 1111 =   -1  (same bits as 255 unsigned)
 0x00 = 0000 0000 =    0

// Same bits, different interpretation:
uint8_t a = 0xFF;   // a = 255
 int8_t b = 0xFF;   // b = -1    ← same bytes in memory!
// overflow behaviorUnsigned overflow wraps around: 255 + 1 = 0. Signed overflow is technically undefined behavior in C, but in practice wraps too. This is a major source of exploitable bugs.

CPU Architecture & Implementation

A CPU has three main parts: Registers (ultra-fast on-chip storage, no RAM needed), the ALU (Arithmetic Logic Unit — does all math and logic), and the Control Unit (reads instructions and coordinates everything). The most important register is RIP (Instruction Pointer) — it always holds the address of the next instruction to run. Controlling RIP is the goal of most exploits.

Fetch–Decode–Execute Cycle

The CPU runs one loop, billions of times per second. Watch it in action above by clicking STEP CYCLE!

  • FETCH CPU reads instruction from RAM at address in RIP.
  • DECODE Control Unit figures out what `MOV` or `ADD` means.
  • EXECUTE ALU performs math or registers update.
  • WRITE Results saved back, RIP advances to next instruction.

Memory Hierarchy — Why Speed Varies

Not all memory is equal. Faster memory is more expensive, so CPUs use a hierarchy. Understanding this matters for performance and for understanding cache-based side-channel attacks (Spectre, Meltdown).

Level Size Access Time Notes
Registers ~16 × 8B 1 cycle On the CPU die, no address needed
L1 Cache 32–64 KB 4 cycles Per-core, split I-cache + D-cache
L2 Cache 256 KB–1 MB 12 cycles Per-core, unified
L3 Cache 4–64 MB 40 cycles Shared between cores
RAM (DRAM) 4–64 GB ~200 cycles Main memory — ASLR randomizes this
SSD/NVMe TB ~100,000 cycles Persistent storage
HDD TB ~10 million cycles Slow — avoid in hot paths
// cache timing attacksMeltdown and Spectre work by measuring how long memory accesses take. If a cache line is hot (recently accessed), it loads in ~4 cycles. If cold, ~200 cycles. An attacker can infer secret data from these timing differences.

Operating System, Processes & Isolation

The OS kernel is the layer between your program and the hardware. It manages processes, memory, files, and network. A process is an isolated program instance — it has its own memory, file descriptors, and state. Two processes cannot touch each other's memory (normally), because the kernel enforces this via the MMU (Memory Management Unit).

When you run ./program, the kernel:

what happens when you run a binary1. Opens and reads the ELF file header
2. Creates a new address space (new set of page tables)
3. Maps program segments into memory:
   .text   → readable + executable  (r-x)
   .rodata → readable only          (r--)
   .data   → readable + writable    (rw-)
   .bss    → readable + writable    (rw-, zeroed)
4. Allocates a stack (grows downward)
5. Allocates initial heap region
6. Loads shared libraries (libc.so, etc.)
7. Sets up environment variables and argv on stack
8. Transfers control to the entry point (_start)
9. _start → __libc_start_main → main()

// View a process's memory map:
cat /proc/1234/maps     # replace 1234 with actual PID
// Or in GDB/pwndbg: vmmap

Virtual Memory — Why Each Process Has Its Own "World"

Every process thinks it owns all of memory from address 0 to 0x7fffffffffff. But two processes can both have a variable at address 0x7fff1234 — they hold completely different data. The CPU's MMU translates virtual addresses to physical RAM via page tables managed by the kernel.

// ASLR — address space layout randomizationThe kernel randomizes where segments are loaded each time you run a program. That's why exploit writers need an "info leak" — a way to read a pointer from the process — before they can reliably overwrite return addresses.

Endianness — Byte Order in Memory

x86/x64 is little-endian: the least-significant byte of a multi-byte value is stored at the lowest memory address. This feels backwards when reading memory dumps. When writing exploit payloads with 64-bit addresses, you must reverse the byte order.

endianness exampleValue to store: 0xDEADBEEF (32-bit integer)

Little-endian (x86/x64) — LSB at lowest address:
  address+0: EF    ← least significant byte
  address+1: BE
  address+2: AD
  address+3: DE    ← most significant byte

Big-endian (network byte order) — MSB at lowest address:
  address+0: DE
  address+1: AD
  address+2: BE
  address+3: EF

// In GDB: x/4xb &var  shows bytes low-to-high
// You see: ef be ad de  — looks "backwards"

ELF Binary Format

On Linux, compiled programs are stored as ELF (Executable and Linkable Format) files. The ELF header tells the OS where to load the program and where execution begins. You'll constantly work with ELF files when doing RE.

ELF structure overviewELF Header (64 bytes):
  Magic:        7f 45 4c 46  ("ELF")
  Class:        64-bit (2) or 32-bit (1)
  Endianness:   little (1) or big (2)
  OS/ABI:       Linux (0)
  Type:         EXEC=executable, DYN=shared lib/PIE, REL=object file
  Machine:      x86-64 (0x3E)
  Entry point:  address of _start()
  PHoff:        offset to Program Header Table
  SHoff:        offset to Section Header Table

Program Headers (what the loader needs):
  LOAD  r-x  offset=0x0000  vaddr=0x400000  size=0x...   ← .text
  LOAD  rw-  offset=0x...   vaddr=0x601000  size=0x...   ← .data/.bss
  DYNAMIC                                                  ← shared lib info
  GNU_STACK  rw-  (NX bit — stack is NOT executable)

// Useful commands:
file ./binary           # what kind of ELF?
readelf -h ./binary     # ELF header
readelf -S ./binary     # section headers
readelf -l ./binary     # program headers (loader segments)
objdump -d ./binary     # disassemble
nm ./binary             # symbol table
strings ./binary        # printable strings
checksec ./binary       # show NX/PIE/Canary/RELRO status
// GOT & PLT — how shared libraries workThe Global Offset Table (GOT) is a writable table of addresses for library functions (printf, malloc, etc.). When printf is called for the first time, the PLT stub jumps to the dynamic linker, which resolves the real address and writes it into the GOT. After that, calls go directly. Overwriting GOT entries is a classic exploitation technique.
