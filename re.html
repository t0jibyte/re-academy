Reverse Engineering

Reverse engineering is the art of understanding software without source code. You take a compiled binary, disassemble it, and figure out exactly what it does. This is the core skill of CTF exploitation, malware analysis, and finding vulnerabilities in closed-source software.

01 — The RE Mindset

Reverse engineering isn't memorizing instructions — it's asking the right questions. Every binary has a purpose. Your job is to find it through systematic analysis, not brute-force reading.

The workflow is always the same: start broad, narrow down. Don't disassemble randomly. Use static analysis to understand structure, then dynamic analysis to observe behavior. Only go to instruction-level detail when you need to.

// the three questionsEvery time you look at a binary, ask: What data does it read? (inputs, files, env), What decisions does it make? (branches, comparisons), and What does it output? (writes, network, exit codes). Everything else is implementation detail.
static vs dynamic analysisStatic  = examining the binary without running it
        → objdump, readelf, Ghidra, strings, file
        → safe for malware, reveals full code, misses runtime behavior

Dynamic = running the binary and observing it
        → GDB, strace, ltrace, ptrace
        → sees real values, follows actual execution path, triggers defenses

Most RE workflows alternate: static to understand structure,
dynamic to verify theories, static again to confirm what you saw.

05 — GOT and PLT — How Library Calls Work

When a binary calls printf, it doesn't know at compile time where printf is in memory — libc loads at a different address every run (ASLR). The solution is a two-table indirection: the PLT and the GOT.

PLT / GOT mechanismWhen your code calls printf:

1. call printf@plt          ; calls the PLT stub (fixed address in binary)

2. PLT stub does:
   jmp [printf@got]         ; indirect jump through GOT entry

3. First call: GOT entry points back to PLT resolver
   → dynamic linker resolves printf's real address
   → writes real address into GOT entry
   → jumps to printf

4. Subsequent calls: GOT entry already has real address
   → direct jump, no resolution needed

; In the disassembly it looks like:
0x401030 <printf@plt>:
    jmp  QWORD PTR [rip+0x200fe2]  ; [0x602018] = printf@got.plt
    push 0                          ; index for lazy resolver
    jmp  0x401000                   ; jump to resolver

# Show GOT entries at runtime with GDB:
(gdb) got                          ; pwndbg command
(gdb) x/gx 0x602018               ; manual: read 8 bytes at GOT entry
→ 0x7ffff7a3d670 <printf>          ; after resolution: real libc address
// why this matters for exploitationWith Partial RELRO (the default), the GOT is writable at runtime. If you can write an arbitrary 8 bytes anywhere (write-what-where), you can overwrite a GOT entry. Then the next time the program calls that function, it jumps to your controlled address instead — giving you code execution.

06 — GDB Workflow — The Essential Commands

GDB core workflow# Launch
$ gdb ./binary
$ gdb -q ./binary          ; quiet (no banner)
$ gdb ./binary core        ; post-mortem: load with coredump

# Run the program
(gdb) run                  ; run with no args
(gdb) run arg1 arg2        ; run with args
(gdb) run < input.txt      ; run with stdin redirect

# Breakpoints
(gdb) break main           ; break at function name
(gdb) break *0x401136      ; break at exact address
(gdb) break *main+42       ; break at offset from symbol
(gdb) info breakpoints     ; list all breakpoints
(gdb) delete 1             ; delete breakpoint #1
(gdb) disable 1            ; disable without deleting

# Stepping
(gdb) next       ; step over (execute one C line)
(gdb) step       ; step into function calls
(gdb) nexti      ; step over ONE assembly instruction
(gdb) stepi      ; step into call (assembly level)
(gdb) continue   ; run until next breakpoint
(gdb) finish     ; run until current function returns
(gdb) until *0x401170  ; run until specific address

# Registers
(gdb) info registers       ; show all registers
(gdb) print $rax           ; print one register
(gdb) print/x $rax         ; print in hex
(gdb) set $rax = 0         ; MODIFY a register

# Memory inspection (x command = examine)
(gdb) x/10gx $rsp          ; 10 × 8-byte hex (stack)
(gdb) x/20wx 0x601000      ; 20 × 4-byte hex (data section)
(gdb) x/s 0x402010         ; as null-terminated string
(gdb) x/i $rip             ; disassemble instruction at RIP
(gdb) x/20i main           ; disassemble 20 instructions from main

# Disassembly
(gdb) disas main           ; disassemble whole function
(gdb) disas 0x401136,+50  ; disassemble 50 bytes from address
(gdb) set disassembly-flavor intel  ; use Intel syntax (do this first!)
(gdb) layout asm           ; TUI: split window with asm view
(gdb) layout regs          ; TUI: add register window

07 — GDB Advanced Techniques

GDB power features# Watchpoints: break when memory changes
(gdb) watch *0x601080      ; break when this address is written
(gdb) rwatch *0x601080     ; break when read
(gdb) awatch *0x601080     ; break on read OR write

# Conditional breakpoints
(gdb) break loop if $rcx == 0

08 — Ghidra Basics

Ghidra is the NSA's open-source decompiler and disassembler. It takes a stripped binary and produces readable C-like pseudocode. It's free, powerful, and the standard tool in CTF and professional RE.

Ghidra workflow1. Open Ghidra → New Project → Import binary
2. Double-click to analyze (accept all default analyzers)
3. Wait for analysis (30s to a few minutes for large binaries)

Key windows:
  Symbol Tree    — lists all functions (main, sub_401234, etc.)
  Decompiler     — shows C-like pseudocode for selected function
  Listing        — disassembly with addresses and bytes
  Data Type Mgr  — struct definitions you can apply to memory

Navigation:
  G           — go to address
  Ctrl+F      — search
  Ctrl+L      — go to label/symbol
  Double-click — follow reference (jump to function/data)
  Alt+←       — go back (like browser back button)

Key operations:
  Rename function:   press L on function name → "check_password"
  Rename variable:   press L on variable in decompiler
  Retype variable:   press Y in decompiler → change int to char*
  Apply struct:      right-click → Data → choose struct type

Finding main in stripped binary:
  → Look for function that calls __libc_start_main
  → First argument to __libc_start_main IS main
  → In Symbol Tree: search for "entry" — this is _start
  → _start calls __libc_start_main(main, argc, argv, ...)

Finding interesting functions fast:
  → Window → Function Call Graph
  → Look for functions called many times (likely helpers)
  → Look for functions calling strcmp, memcmp (often password checks)
  → Search strings → double-click → "references to" → lands in code

09 — Reading Decompiled C

Ghidra's decompiler output isn't real C — it's a best-effort translation. Learning to read it means recognizing what real code looks like after compilation and decompilation artifacts.

decompiler output vs real C// Ghidra decompiler output (stripped, -O2)
undefined8 FUN_00401136(void)
{
  char *pcVar1;
  long in_FS_OFFSET;
  char local_48 [64];
  long local_10;

  local_10 = *(long *)(in_FS_OFFSET + 0x28);    // stack canary read
  printf("Enter password: ");
  fgets(local_48, 64, stdin);
  pcVar1 = strchr(local_48, '\n');              // strip newline
  if (pcVar1 != (char *)0x0) {
    *pcVar1 = '\0';
  }
  pcVar1 = (char *)strcmp(local_48, "s3cr3t_p4ss");
  if (pcVar1 == (char *)0x0) {                  // strcmp returns 0 on match
    puts("Correct!");
  }
  else {
    puts("Wrong!");
  }
  if (local_10 != *(long *)(in_FS_OFFSET + 0x28)) {  // canary check
                    /* WARNING: Subroutine does not return */
    __stack_chk_fail();
  }
  return 0;
}

// What this actually was in the original source:
int main() {
  char buf[64];
  printf("Enter password: ");
  fgets(buf, 64, stdin);
  buf[strcspn(buf, "\n")] = 0;
  if (strcmp(buf, "s3cr3t_p4ss") == 0)
    puts("Correct!");
  else
    puts("Wrong!");
  return 0;
}

// Key decompiler artifacts to recognize:
// - in_FS_OFFSET = thread-local storage pointer → stack canary
// - undefined8 = 8-byte return value (could be int, long, pointer)
// - local_10, local_48 = named by stack offset, not original names
// - (char *)0x0 = NULL
// - strcmp cast to char* = Ghidra confused about return type

10 — Recognizing Common Binary Patterns

patterns in disassembly// 1. String comparison — the CTF password check
0x401156: lea  rsi, [rip+0x1aa]   ; "s3cr3t"
0x40115d: mov  rdi, rax            ; user input
0x401160: call strcmp@plt
0x401165: test eax, eax
0x401167: jne  0x401180            ; if not equal: fail
; Cheat: set $eax = 0 in GDB to force "equal" and bypass

// 2. Anti-debug: ptrace check (detects GDB)
0x401200: mov  eax, 101            ; syscall 101 = ptrace
0x401205: xor  edi, edi            ; PTRACE_TRACEME
0x401207: xor  esi, esi
0x401209: xor  edx, edx
0x40120b: xor  r10d, r10d
0x40120e: syscall
0x401210: cmp  rax, -1
0x401214: je   0x401250            ; if ptrace fails → being debugged
; Bypass: set $rax = 0 in GDB right after syscall

// 3. XOR obfuscation (decrypting a string at runtime)
0x401300: lea  rdi, [rip+0x200]   ; encrypted bytes
0x401307: mov  ecx, 16            ; length
0x40130c: mov  al, [rdi+rcx-1]
0x401310: xor  al, 0x42           ; XOR key
0x401312: mov  [rdi+rcx-1], al
0x401316: dec  ecx
0x401318: jnz  0x40130c
; Trick: break after the loop, x/s $rdi to read decrypted string

// 4. Stack canary check
0x4011a0: mov  rax, [rbp-0x8]     ; load saved canary
0x4011a4: xor  rax, [fs:0x28]     ; compare with original
0x4011ad: jne  0x4011bf           ; mismatch → crash
0x4011af: leave
0x4011b0: ret

// 5. A vtable call (C++)
0x401250: mov  rax, [rdi]         ; load vptr from object
0x401253: mov  rax, [rax+0x10]   ; vtable[2] = third virtual function
0x401257: call rax               ; dispatch

12 — Anti-RE Tricks and Bypasses

Technique How to Detect Bypass
ptrace() anti-debug syscall 101 or ptrace() call near start GDB: set $rax=0 after syscall; or patch the binary
Timing checks gettimeofday/clock_gettime before and after, compare delta Step over the check; patch je→jmp
Self-modifying code mprotect() then writes to .text Set breakpoint AFTER modification, read decrypted code
XOR string obfuscation Strings aren't readable in .rodata; loop with xor before use Break after deobfuscation loop, dump memory
UPX packing Tiny .text, large entropy, "UPX" string upx -d ./binary or break at OEP
Stripped symbols file says "stripped"; nm shows no T symbols Ghidra analysis; identify main via __libc_start_main
Opaque predicates Unreachable code, always-true conditions Recognize constant-expression branches; ignore dead paths 
patch a binary to bypass a check# Turn a conditional jump into unconditional (NOP it out)
# Find the je instruction that skips our code:
$ objdump -d -M intel ./binary | grep -B5 "Wrong"
  401167: 74 17       je  0x401180    ← 74 17 = je rel8

# NOP out the jump (replace with 0x90 0x90)
$ printf '\x90\x90' | dd of=./binary bs=1 seek=$((0x167)) conv=notrunc

# OR: flip je (0x74) to jne (0x75) — takes opposite branch
$ printf '\x75' | dd of=./binary bs=1 seek=$((0x167)) conv=notrunc

# Make it executable and test
$ chmod +x ./binary && ./binary
